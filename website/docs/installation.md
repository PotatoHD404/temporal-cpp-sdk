# Installing & packaging

[Getting started](getting-started.md) covers the macOS quick-path. This page is the cross-platform
build, install, and packaging reference: how to get the toolchain dependencies on each OS, the build
options, and how to consume the installed SDK from another project.

## Requirements

- A C++20 compiler — recent Clang/AppleClang, GCC ≥ 11, or MSVC 19.3x
- CMake ≥ 3.21
- The dependencies **gRPC**, **Protobuf** (C++), and **nlohmann-json**
- The `temporalio/api` protobuf definitions, vendored as a git submodule and compiled at build time:

  ```bash
  git submodule update --init third_party/api
  ```

GoogleTest is fetched automatically (`FetchContent`) unless a `GTest` config package is already
present (e.g. from Conan or a system install).

## Get the dependencies

You can bring the C++ toolchain deps in three ways. Pick whichever fits your platform.

### macOS — Homebrew

```bash
brew install cmake grpc protobuf nlohmann-json
```

CMake finds the Homebrew copies automatically (the build prepends the Homebrew prefix to
`CMAKE_PREFIX_PATH` — but only when no toolchain file is supplied, so a Conan toolchain is never
shadowed).

### Linux — system packages

Debian/Ubuntu:

```bash
sudo apt-get install -y cmake make g++ \
  libgrpc++-dev libprotobuf-dev protobuf-compiler protobuf-compiler-grpc \
  nlohmann-json3-dev
```

Fedora:

```bash
sudo dnf install cmake gcc-c++ grpc-devel grpc-plugins protobuf-devel json-devel
```

### Any platform — Conan

[Conan](https://conan.io) resolves the deps from source/binaries on macOS, Linux, and Windows, so
you don't depend on a system package manager:

```bash
pip install conan
conan profile detect --force
git submodule update --init third_party/api

conan install . --build=missing      # resolves gRPC + Protobuf + nlohmann_json
cmake --preset conan-release         # preset generated by Conan
cmake --build --preset conan-release
```

:::note
The `conanfile.py` pins a known-compatible gRPC/Protobuf pair. Bump them together if you need a
newer toolchain — gRPC is built against a specific Protobuf.
:::

## Build

With the dependencies in place, the build is the same everywhere:

```bash
cmake --preset default          # configure (generates protobuf C++, fetches GoogleTest)
cmake --build build -j          # build the library, example, and tests
ctest --test-dir build -LE integration   # unit tests (no server needed)
```

### Build options

| Option | Default | Effect |
| --- | --- | --- |
| `TEMPORAL_BUILD_TESTS` | `ON` | Build the unit + integration test driver. |
| `TEMPORAL_BUILD_EXAMPLES` | `ON` | Build the bundled examples. |
| `TEMPORAL_INSTALL` | `ON` | Emit `install()` rules and the `find_package(temporal-cpp)` package config. |
| `TEMPORAL_ENABLE_CLANG_TIDY` | `OFF` | Run clang-tidy on first-party sources during the build. |

Example — a lean library-only build for packaging:

```bash
cmake --preset default -DTEMPORAL_BUILD_TESTS=OFF -DTEMPORAL_BUILD_EXAMPLES=OFF
```

## Install

```bash
cmake --install build --prefix /your/prefix
```

This installs the static archives (`temporal_sdk`, `temporal_proto`), the public headers under
`include/temporal/…` (plus the generated protobuf headers), and a CMake package config under
`lib/cmake/temporal-cpp/`.

## Consume it from another project

### With CMake `find_package`

```cmake
cmake_minimum_required(VERSION 3.20)
project(my_app CXX)

find_package(temporal-cpp CONFIG REQUIRED)

add_executable(my_app main.cpp)
target_link_libraries(my_app PRIVATE temporal::sdk)
```

Point CMake at the install tree (or build tree) and the transitive deps:

```bash
cmake -S . -B build -DCMAKE_PREFIX_PATH="/your/prefix"
```

The package config re-finds gRPC, Protobuf, nlohmann_json, and Threads for you, so a consumer only
needs to link `temporal::sdk`. (`tests/packaging/` in this repo is exactly this consumer, used as a
CI smoke test of the install.)

### With Conan

If you publish the recipe (`conan create . --build=missing`), a consumer can depend on it and get
the same `temporal::sdk` target:

```python
# conanfile.txt
[requires]
temporal-cpp/0.1.0

[generators]
CMakeDeps
CMakeToolchain
```

## Platform support

| Platform | How | Status |
| --- | --- | --- |
| macOS (arm64) | Homebrew or Conan | Built + fully tested in CI |
| Linux (x86-64) | apt or Conan | Built + unit-tested in CI |
| Windows (MSVC) | Conan | Compiler flags are MSVC-safe; not yet exercised in CI |

The CMake setup avoids platform-specific assumptions (warning flags are gated to GNU/Clang, the
`protoc`/`grpc_cpp_plugin` tools are resolved from imported targets so they come from whatever
provided the libraries). Windows is expected to work through Conan but is not yet covered by CI.
