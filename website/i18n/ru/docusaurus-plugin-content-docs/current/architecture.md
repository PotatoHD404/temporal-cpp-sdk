# Архитектура

`temporal-cpp-sdk` — это **нативный** SDK: он напрямую взаимодействует с frontend-сервером Temporal через gRPC и самостоятельно реализует движок реплея воркфлоу, критичный к детерминизму, отражая структуру Go SDK. Зависимости от Rust `sdk-core` нет.

## Слои

```
include/temporal/        Публичный API (без proto)
  client/  worker/  workflow/  activity/  converter/  common/  log/
src/
  client/  worker/        Тонкие публичные реализации
  converter/  log/        Конвертация данных, структурированное логирование
  internal/               Движок (никогда не попадает в публичные заголовки):
    grpc_client           Обёртка стаба WorkflowService
    coroutine             Стекозависимая поточная корутина (fiber)
    workflow_task_handler Движок реплея, sticky-кэш, runner
    activity_task_handler Выполнение активностей + heartbeat
    worker_impl           Потоки поллера + жизненный цикл
    proto_util            Payload/Failure/Duration <-> protobuf, идентификаторы
```

Жёсткое правило: **публичные заголовки никогда не включают protobuf** — вся gRPC/proto-поверхность находится исключительно в `src/internal/`.

## Проблема детерминизма

Воркфлоу должен быть **детерминированным и воспроизводимым**: история событий является источником истины, и SDK обязан восстановить точное состояние воркфлоу в памяти, реплицируя эту историю. Каждый оркестрационный вызов перехватывается, преобразуется в *команду* для сервера и впоследствии сопоставляется с соответствующими событиями истории. Это наиболее сложная, нативная часть любого SDK.

## Диспетчер стекозависимых корутин

Тело воркфлоу выполняется на **стекозависимой корутине** — кооперативном потоке-файбере, управляемом строгим пошаговым протоколом передачи управления, при котором диспетчер и поток воркфлоу никогда не работают одновременно. (Это C++-аналог диспетчера на основе горутин, который использует Go SDK.)

Ожидание неразрешённого future или сигнала **приостанавливает** корутину — сохраняя весь стек воркфлоу и локальные переменные — вместо его раскрутки. Именно это позволяет **запросам (query)** читать живое состояние, а **селекторам** выбирать среди активных future. Завершение (вытеснение из кэша или завершение воркфлоу) корректно раскручивает приостановленный фрейм.

## Sticky-кэш

Корутина выполняющегося воркфлоу хранится резидентно между задачами, с ключом по run id. Воркер объявляет уникальную **sticky-очередь задач** в каждом ответе и опрашивает её вторым поллером, поэтому сервер доставляет задачи продолжения, несущие только **инкрементальную** историю:

- **Продолжение** (задача продолжает кэшированное состояние): применяются только новые события к живым future / состоянию сигналов, затем возобновляется корутина — **без полного реплея**.
- **Реплей** (первая задача или история не продолжает кэш): восстановление из полной истории, выполнение до текущей точки приостановки и кэширование runner.
- **Промах sticky-кэша** (инкрементальная история, но ничего не кэшировано): возвращается `RESET_STICKY_TASK_QUEUE`, чтобы сервер переслал полную историю.

Кэш представляет собой ограниченный LRU: `WorkerOptions::max_cached_workflows` ограничивает резидентный набор (`0` = без ограничения), и при заполнении вытесняется наименее недавно использованный runner — его следующая задача реплеится из полной истории. `Worker::cache_hits()` / `replays()` предоставляют разбивку «продолжение против реплея».

## Активации → команды

Каждая задача передаёт производное из истории состояние выполняющемуся воркфлоу, который порождает команды
(`ScheduleActivityTask`, `StartTimer`, `StartChildWorkflowExecution`, `CompleteWorkflowExecution`,
`ContinueAsNewWorkflowExecution`, …). Обновления (update) передаются как протокольные сообщения
(`Acceptance` / `Response`). Это та же модель активация→команда, которую используют SDK на основе core, только здесь реализованная нативно на C++.

## Повторное применение обновлений при реплее {#replay-re-application-of-updates}

При sticky-**продолжении** движок просто применяет новые события к живым future и возобновляет работу, поэтому состояние, изменённое обновлением, уже находится в памяти. Реплей **с нуля** (холодный старт, вытеснение из sticky-кэша или `Worker::ReplayWorkflowHistory`) сложнее: предсканирование заранее разрешает каждый известный из истории future, поэтому одно наивное возобновление обогнало бы тело воркфлоу до завершения *прежде*, чем перезапустится хоть один принятый обработчик обновления, — и тело никогда не увидело бы состояние, заданное обновлением.

Чтобы воссоздать живое чередование, предсканирование разбивает историю на **первом принятом обновлении**. Всё, что до него, ускоренно проигрывается начальным возобновлением как обычно. Всё, что *после* него, — каждое разрешение future и доставленный сигнал — **откладывается** в `post_update_timeline` вместо предварительного применения. Затем движок проигрывает эту хронологию в **порядке истории**, вставляя принятые обновления обратно на их записанные позиции: каждое обновление перезапускает свой обработчик, каждое отложенное событие разрешает future или доставляет сигнал, а возобновление после каждого шага позволяет телу продвинуться до следующей точки парковки. Состояние, изменённое обновлением, — флаг, который тело ожидает через `Await`, счётчик, который завершает более поздний сигнал, — восстанавливается ровно таким, каким оно было вживую.

Поскольку испускаемые команды защищены от повторного срабатывания, перезапуск обработчика или повторное разрешение future никогда не перепланируют работу; они лишь продвигают тело и порядок порождаемых команд, с которым сравнивает проверка детерминизма. **Маркеры / побочные эффекты *не* откладываются** — они потребляются в порядке вызова, независимо от тайминга событий. Для распространённого воркфлоу без обновлений хронология пуста, поэтому реплей остаётся ускоренной перемоткой одним возобновлением, без изменений.

## Прерывание при дедлоке

Код воркфлоу никогда не должен блокироваться. Диспетчер возобновляет каждую корутину под необязательным ограничением на одно возобновление (`WorkerOptions::deadlock_detection_timeout`): возобновление выполняется через `ResumeFor(timeout)`, и задача, не уступившая управление в его пределах, помечает runner как заблокированный (deadlocked). Тогда обработчик **прерывает выполнение** — он испускает метрику `temporal_workflow_task_deadlock`, логирует ошибку, удаляет запуск из кэша и завершает *задачу* воркфлоу с ошибкой, чтобы сервер перепланировал её.

Сорвавшуюся корутину нельзя безопасно прервать, поэтому она **покидается**: она продолжает работать на собственном отсоединённом потоке, а её runner перемещается в список утечек, живущий всё время процесса, чтобы этот поток никогда не разыменовывал освобождённую память. Поток поллера немедленно возвращается и продолжает обслуживать другие воркфлоу. Это тот же компромисс, на который идёт Go SDK (горутина заблокированного воркфлоу утекает). Операционный взгляд см. в [Запуск в продакшне](/production#deadlock-detection).

## Что намеренно отсутствует (пока)

Обнаружение недетерминизма, версионирование воркфлоу (`GetVersion`/patching), побочные эффекты
(`SideEffect`/`MutableSideEffect`) и ограниченный LRU sticky-кэш (`max_cached_workflows`) — некогда
перечисленные здесь как будущая работа — теперь реализованы (см. [Sticky-кэш](#sticky-кэш) и
[матрицу паритета](/parity)). Остальные пробелы на уровне движка (TLS, полное испускание метрик/трейсов,
маршрутизация версионирования воркеров, кодеки payload, …) отслеживаются в [матрице паритета](/parity#roadmap).
Файл [`docs/ARCHITECTURE.md`](https://github.com/) в репозитории содержит ту же схему на уровне файлов и символов.
