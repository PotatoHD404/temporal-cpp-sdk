# temporal-cpp

**Нативный C++ SDK для [Temporal](https://temporal.io).** Взаимодействует с фронтендом Temporal
по gRPC и самостоятельно выполняет движок реплея воркфлоу — смоделирован по официальному
[Go SDK](https://github.com/temporalio/sdk-go), без зависимости от Rust-библиотеки `sdk-core`.

```cpp
#include <temporal/temporal.h>

std::string Greet(temporal::workflow::Context& ctx, std::string name) {
  temporal::ActivityOptions o;
  o.start_to_close_timeout = std::chrono::seconds(10);
  return ctx.ExecuteActivity<std::string>(o, "ComposeGreeting", name).Get();
}
```

:::warning Экспериментально — не является официальным продуктом Temporal
Это сообщественный проект, **не** являющийся официальным Temporal SDK. Он реализует
существенное, полностью протестированное **ядро** модели программирования Temporal, однако
**не находится в паритете** с официальными SDK. См. [Возможности и паритет](/parity) — честную
подробную матрицу того, что работает, а что нет.
:::

## Почему этот проект существует

Официального C++ SDK для Temporal не существует. SDK Temporal делятся на два вида:

- **Нативные** (Go, Java) — реализуют gRPC-клиент *и* критичный для детерминизма автомат
  состояний воркфлоу / движок реплея истории событий непосредственно на языке хоста.
- **На основе Core** (Python, TypeScript, .NET, Ruby) — делегируют этот движок Rust-библиотеке
  [`sdk-core`](https://github.com/temporalio/sdk-core).

`temporal-cpp` идёт по **нативному** пути, воспроизводя эргономику Go SDK в идиоматичном C++20
поверх собственного движка (диспетчер на стекозависимых корутинах со sticky-кэшем в памяти).

## Что работает сегодня

- **Клиент** — подключение, запуск воркфлоу, ожидание результатов (с поддержкой continue-as-new),
  сигналы, запросы (query), обновления (update), отмена, завершение.
- **Воркер** — регистрация обычных функций вида `R(Context&, Args...)`, потоки опроса, sticky-кэш.
- **Активности** — типизированное выполнение, управляемые сервером повторы с настраиваемой
  `RetryPolicy`, heartbeat.
- **Таймеры** — `NewTimer` / `Sleep`.
- **Передача сообщений воркфлоу** — сигналы, запросы (query), обновления (update).
- **Композиция** — селекторы («активность ИЛИ таймаут»), дочерние воркфлоу, continue-as-new,
  отмена в режиме наблюдения.
- **Движок** — диспетчер корутин, сохраняющий живое состояние воркфлоу между приостановками;
  sticky-кэш, благодаря которому задачи продолжения применяют только инкрементальную историю
  (без полного ре-реплея).
- **Тесты** — 18 юнит-тестов + 17 сквозных интеграционных тестов против реального dev-сервера
  Temporal, а также CI.

## Чего не хватает

Многого — достижение полного паритета — это труд на несколько лет. Явно отсутствуют:
версионирование/патчинг воркфлоу, side effects, фреймворк тестирования/реплея, расписания,
Nexus, версионирование воркеров, интерцепторы, TLS/mTLS и аутентификация по API-ключу,
метрики/трейсинг, поисковые атрибуты и memo, локальные активности, payload-кодеки, а также
значительная часть поверхности operator/cloud-клиента. Всё это перечислено в
[матрице паритета](/parity).
