# temporal-cpp-sdk

**Нативный C++ SDK для [Temporal](https://temporal.io).** Взаимодействует с фронтендом Temporal
по gRPC и самостоятельно выполняет движок реплея воркфлоу — смоделирован по официальному
[Go SDK](https://github.com/temporalio/sdk-go), без зависимости от Rust-библиотеки `sdk-core`.

```cpp
#include <temporal/temporal.h>

std::string Greet(temporal::workflow::Context& ctx, std::string name) {
  temporal::ActivityOptions o;
  o.start_to_close_timeout = std::chrono::seconds(10);
  return ctx.ExecuteActivity<std::string>(o, "ComposeGreeting", name).Get();
}
```

:::warning Экспериментально — не является официальным продуктом Temporal
Это сообщественный проект, **не** являющийся официальным Temporal SDK. Он реализует
существенное, полностью протестированное **ядро** модели программирования Temporal, однако
**не находится в паритете** с официальными SDK. См. [Возможности и паритет](/parity) — честную
подробную матрицу того, что работает, а что нет.
:::

## Почему этот проект существует

Официального C++ SDK для Temporal не существует. SDK Temporal делятся на два вида:

- **Нативные** (Go, Java) — реализуют gRPC-клиент *и* критичный для детерминизма автомат
  состояний воркфлоу / движок реплея истории событий непосредственно на языке хоста.
- **На основе Core** (Python, TypeScript, .NET, Ruby) — делегируют этот движок Rust-библиотеке
  [`sdk-core`](https://github.com/temporalio/sdk-core).

`temporal-cpp-sdk` идёт по **нативному** пути, воспроизводя эргономику Go SDK в идиоматичном C++20
поверх собственного движка (диспетчер на стекозависимых корутинах со sticky-кэшем в памяти).

## Что работает сегодня

- **Клиент** — подключение, запуск воркфлоу, ожидание результатов (с поддержкой continue-as-new),
  сигналы, запросы (query), обновления (update), отмена, завершение; расписания (интервал + cron),
  пакетные операции, visibility-запросы, управление поисковыми атрибутами и асинхронное завершение
  активностей (`CompleteActivity`/`FailActivity`).
- **Воркер** — регистрация обычных функций вида `R(Context&, Args...)` (или типизированных
  дескрипторов через `TEMPORAL_ACTIVITY`), потоки опроса, sticky-кэш и реплей истории
  (`ReplayWorkflowHistory`).
- **Активности** — типизированное выполнение, управляемые сервером повторы с настраиваемой
  `RetryPolicy`, троттлинг heartbeat, локальные активности и асинхронное завершение.
- **Таймеры** — `NewTimer` / `Sleep` (а также `chrono`-литералы вроде `10s` / `24h`).
- **Передача сообщений воркфлоу** — сигналы, запросы (query), обновления (update, с валидаторами),
  все они также адресуются через типизированные дескрипторы `SignalRef`/`QueryRef`/`UpdateRef`.
- **Помощники детерминизма** — `SideEffect` / `MutableSideEffect`, `GetVersion` (патчинг),
  `UpsertSearchAttributes`.
- **Композиция** — селекторы («активность ИЛИ таймаут»), дочерние воркфлоу (с
  `ParentClosePolicy` + сигналом дочернему), continue-as-new, отмена в режиме наблюдения и
  Nexus-операции.
- **Движок** — диспетчер корутин, сохраняющий живое состояние воркфлоу между приостановками;
  sticky-кэш, благодаря которому задачи продолжения применяют только инкрементальную историю
  (без полного ре-реплея).
- **Тестирование** — time-skipping-окружение `TestWorkflowEnvironment` и
  `Worker::ReplayWorkflowHistory` для юнит-тестирования воркфлоу против записанной истории.
- **Протестировано** — 86 юнит-тестов + сквозной интеграционный набор против реального dev-сервера
  Temporal, а также CI.

## Чего не хватает

Достижение полного паритета — это всё ещё труд на несколько лет. Оставшиеся пробелы в основном
лежат на периферии: встроенные экспортеры метрик/трейсинга (приёмник `MetricsHandler` и хуки
трейсинга на месте, но экспортер OTel не поставляется), охват payload-кодеков сверх встроенных
base64/gzip, закрепление версии на стороне воркера и часть поверхности operator/cloud-клиента,
требующая инфраструктуры, которую один dev-сервер предоставить не может.
[Матрица паритета](/parity) перечисляет всё это, с оговорками.
