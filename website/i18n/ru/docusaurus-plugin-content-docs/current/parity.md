# Возможности и паритет

**Данный SDK не находится в паритете с официальными Temporal SDK и в ближайшее время им не станет.** Официальные SDK для Go/Java/Python/TS/.NET — это годы работы целых команд; один только движок Temporal на Go занимает десятки тысяч строк, а полная поверхность API (расписания (schedules), Nexus, версионирование, фреймворк для тестирования, безопасность, наблюдаемость, operator/cloud-клиент…) огромна.

Что представляет собой этот проект: подлинно работающее, **полностью протестированное ядро** модели программирования Temporal — достаточное для написания и запуска реальных воркфлоу, которые оркестрируют активности, реагируют на сигналы/запросы (query)/обновления (update), компонуют дочерние воркфлоу и селекторы, а также выполняют continue-as-new — на нативном движке C++ со sticky-кэшем. Эта страница — честная картина текущего состояния.

**Условные обозначения:** ✅ реализовано и протестировано · 🟡 частично · ❌ не реализовано

| Возможность | Статус | Примечания |
|---|---|---|
| Диспетчер на stackful-корутинах | ✅ | |
| Sticky-кэш + инкрементальная история | ✅ | |
| Обнаружение недетерминизма | ✅ | воспроизведённые команды сверяются с историей по порядку; `WorkflowPanicPolicy` (блокировать/завершить) |
| Повторное применение обновлений при реплее | ✅ | при реплее с нуля (после вытеснения из кэша) принятые обновления повторно выполняются в записанном чередовании: события, следующие за первым обновлением, откладываются и воспроизводятся в порядке истории с вклинёнными обратно обработчиками обновлений, так что состояние, изменённое обновлением — флаг, которого ждёт тело, счётчик, завершаемый более поздним сигналом — восстанавливается. e2e-протестировано |
| Пагинация истории событий | ✅ | пути workflow-задачи / query / экспорта собирают постраничную историю через `next_page_token` |
| Обнаружение дедлоков / политики паник | ✅ | политики паник ✅ (`WorkflowPanicPolicy`); задача, превысившая `deadlock_detection_timeout`, обнаруживается, репортится (метрика/лог) И прерывается — задача проваливается (сервер перепланирует), воркер продолжает обслуживать другие воркфлоу; зависшая корутина выполняется на своём потоке, который при прерывании отсоединяется и намеренно «течёт» (как Go «теряет» зависшую горутину воркфлоу). e2e-протестировано |

| Расписания (schedules) | ✅ | полный клиентский цикл (create/describe/delete/update/list/trigger/pause) + интервал **и** calendar/cron-спеки |

## Дорожная карта {#roadmap}

Большая часть исходной дорожной карты уже ✅ (см. матрицу выше):

1. **Упрочение детерминизма** — ✅ обнаружение недетерминизма + ограниченный LRU sticky-кэш (`max_cached_workflows`) + обнаружение отмены в heartbeat + пагинация истории.
2. **Поверхность возможностей воркфлоу** — ✅ `SideEffect`/`MutableSideEffect` + `GetVersion` + валидаторы обновлений + отмена (таймер, активность, дочерний воркфлоу, `AwaitCancellation`) + случаи channel для селекторов + локальные активности.
3. **Производственные требования** — ✅ TLS/mTLS + аутентификация по API-ключу, интерсепторы, метрики и трейсинг, proto/protoJSON-конвертеры + payload-кодеки, настройка воркеров (лимиты, rate limit, плавный дренаж, автомасштабирование поллеров).
4. **Расширение охвата** — ✅ фреймворк для реплея/тестирования + расписания (полный клиентский цикл) + версионирование воркеров (build-id + правила + деплойменты) + сессии (host-пиннинг) + бо́льшая часть operator-поверхности.

**Недавно закрыто** — каждый постатейный пробел из прежних редакций этой страницы теперь ✅:

- ✅ **Повторное применение обновлений при реплее** — принятые обновления повторно выполняются в записанном чередовании при реплее с нуля (таймлайн отложенных событий; см. строку «Детерминизм»).
- ✅ **Nexus-операции** — команда-вызыватель (`ExecuteNexusOperation`) + worker-обработчик Nexus (`RegisterNexusOperation`), поверх управления эндпоинтами.
- ✅ **Прерывание дедлока** — превысившая лимит задача обнаруживается при возобновлении, репортится (метрика/лог) и прерывается; зависшая корутина выполняется на своём потоке, отсоединяется и «течёт», так что воркер продолжает обслуживать (как Go SDK «теряет» зависшую горутину воркфлоу).
- ✅ **Time-skip в фреймворке тестирования** — `testing::TestWorkflowEnvironment` управляет TestService time-skipping-тест-сервера Temporal (lock/unlock/sleep/get-current-time), так что насыщенные таймерами воркфлоу тестируются за миллисекунды.
- ✅ мелкие детали: троттлинг heartbeat, calendar/cron-спеки расписаний, parent-close-policy + сигнал дочернему, типизированное декодирование ошибок на клиенте.

**Что всё ещё ограничивает паритет** (глубина, а не постатейная дорожная карта):

Постатейная дорожная карта выполнена, но это всё ещё не полная официальная поверхность. Честные оговорки живут в матрице выше, например: TLS/mTLS проверен юнит-тестами, но ещё не запускался против настоящего Temporal-сервера с TLS; часть поверхности operator/версионирования (федерация remote-cluster, версионирование worker-деплойментов) реализована, но требует инфраструктуры, которую один dev-сервер предоставить не может; долгоживущие сессии с heartbeat-удержанием, встроенный экспортер OTel и несколько внутренних метрик движка Go не реализованы. А широкая поверхность, упомянутая во введении (полный охват schedules/Nexus/visibility/безопасности/cloud), по-прежнему отстаёт от официальных SDK на годы работы.

Если возможность показывает оговорку или 🟡/❌ в матрице — её действительно ещё нет в полном объёме. Пожалуйста, не делайте противоположных выводов, опираясь на работающее ядро.
