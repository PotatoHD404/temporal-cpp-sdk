# Запуск в продакшене

На этой странице рассматриваются эксплуатационные аспекты, которые необходимо учесть перед тем, как разворачивать воркер `temporal-cpp-sdk` в продакшн-окружении: как настроить `WorkerOptions`, как наблюдать за sticky-кэшем, как защититься от недетерминистичных изменений кода и что делать с возможностями, которые ещё не реализованы.

:::warning
Несколько критически важных для продакшена функций — TLS/mTLS, аутентификация, метрики, распределённая трассировка и версионирование воркеров — **ещё не реализованы**. Ознакомьтесь с разделом [Ещё не для продакшена](#not-yet-for-production) и полной [матрицей совместимости](/parity) перед деплоем.
:::

## Опции воркера

Передайте структуру `temporal::WorkerOptions` третьим аргументом в конструктор `Worker`:

```cpp
#include <temporal/common/options.h>
#include <temporal/worker/worker.h>

temporal::WorkerOptions opts;
opts.max_concurrent_workflow_tasks = 8;
opts.max_concurrent_activities     = 100;
opts.workflow_task_pollers         = 2;
opts.activity_task_pollers         = 8;
opts.panic_policy                  = temporal::WorkflowPanicPolicy::BlockWorkflow;
opts.max_cached_workflows          = 500;

temporal::worker::Worker worker(client, "my-task-queue", opts);
```

### Справочник по полям

| Поле | Значение по умолчанию | Что контролирует |
|---|---|---|
| `max_concurrent_workflow_tasks` | `0` (значение библиотеки) | Максимальное число задач воркфлоу, одновременно выполняемых на данном воркере. Каждая активная задача занимает стек корутины. |
| `max_concurrent_activities` | `0` (значение библиотеки) | Максимальное число активностей, одновременно выполняемых на данном воркере. |
| `workflow_task_pollers` | `1` | Число потоков-поллеров, выделенных для обычной (нестикованной) очереди задач воркфлоу. |
| `activity_task_pollers` | `1` | Число потоков-поллеров, выделенных для очереди задач активностей. |
| `panic_policy` | `BlockWorkflow` | Реакция при обнаружении расхождения воспроизведённых команд с записанной историей. См. [Защита от недетерминизма](#non-determinism-safety). |
| `max_cached_workflows` | `0` (без ограничений) | Максимальное число корутин воркфлоу, одновременно хранящихся в sticky-кэше. `0` означает отсутствие ограничения. При достижении лимита вытесняется наименее недавно использованный воркфлоу. См. [Sticky-кэш](#sticky-cache). |

### Рекомендации по размерам

**Поллеры задач воркфлоу** (`workflow_task_pollers`): сервер отдаёт задачи воркфлоу по одной на каждый опрос, поэтому двух-четырёх поллеров обычно достаточно даже при высокой нагрузке. При первоначальном деплое часто хватает одного поллера.

**Поллеры задач активностей** (`activity_task_pollers`): I/O-bound-активности выигрывают от большего числа поллеров. Отправной точкой служит `max(1, std::thread::hardware_concurrency() / 2)` с учётом ограничений downstream-зависимостей.

**Параллельные активности** (`max_concurrent_activities`): жёсткий лимит одновременных выполнений активностей. Настройте его с учётом размера пула соединений с базой данных, внешних rate-ограничений или бюджета памяти. `0` означает, что библиотека применяет собственное значение по умолчанию; задайте явное значение, как только определите ресурсный бюджет.

**Параллельные задачи воркфлоу** (`max_concurrent_workflow_tasks`): каждая активная задача воркфлоу удерживает стек корутины в памяти. Для продолжений из кэша затраты невелики; для реплея (холодный старт или вытеснение) они включают повторный прогон полной истории воркфлоу. Начните со значения, близкого к `max_cached_workflows`, и увеличивайте, если видите накопление в sticky-очереди.

:::note
Параметры конкурентности и rate-limiting присутствуют и принимаются, однако **не все полностью соблюдаются** текущей реализацией (см. [матрицу совместимости](/parity)). Рассматривайте их как резервирования, которые библиотека соблюдает по возможности — до завершения полноценного enforcement.
:::

## Sticky-кэш {#sticky-cache}

Воркер держит стекфульную корутину выполняющегося воркфлоу в памяти между задачами, индексируя её по run id. Когда сервер доставляет задачу-продолжение, воркер применяет только инкрементальную историю и возобновляет живую корутину — полный реплей не требуется. Подробности архитектуры см. в разделе [Архитектура](/architecture).

### Ограничение кэша

По умолчанию `max_cached_workflows = 0` означает, что кэш не ограничен и воркфлоу вытесняются только при завершении. В продакшене почти всегда стоит задать конечное ограничение:

```cpp
opts.max_cached_workflows = 500;  // держать не более 500 резидентных корутин
```

Когда кэш заполнен и поступает новая задача для некэшированного запуска, вытесняется **наименее недавно использованная** запись. Стек корутины вытесненного воркфлоу очищается; следующая задача инициирует полный реплей для восстановления состояния. Выбирайте значение, которое комфортно укладывается в бюджет памяти воркера. Резидентная корутина обычно занимает несколько сотен килобайт стека плюс память, выделяемую телом воркфлоу в куче.

:::note
`max_cached_workflows = 0` (без ограничений) безопасно для разработки и малонагруженных деплоев с небольшим числом активных запусков. При любой устойчивой конкурентности задайте ограничение — неограниченный кэш может исчерпать память процесса, если одновременно открыто много запусков воркфлоу.
:::

### Наблюдение за эффективностью кэша

`Worker` предоставляет два счётчика, которые можно опрашивать и отправлять в систему мониторинга:

```cpp
// После Start() опрашивайте их в периодическом потоке или перед Stop():
long hits    = worker.cache_hits();   // задачи, обслуженные из живой корутины (без реплея)
long misses  = worker.replays();      // задачи, потребовавшие полного реплея истории

double hit_rate = (double)hits / std::max(1L, hits + misses);
```

Здоровый воркер в установившемся режиме имеет высокий hit rate (> 95 %). Устойчивый низкий hit rate свидетельствует о том, что кэш слишком мал для числа одновременно открытых запусков, воркер часто перезапускается или очередь задач обслуживается пулом воркеров без аффинности (каждый воркер воспроизводит задачи, закэшированные у другого).

## Защита от недетерминизма {#non-determinism-safety}

Воркфлоу детерминирован: воспроизведение его полной истории должно давать в точности ту же последовательность оркестрационных команд. Если вы деплоите изменение кода, которое переставляет, удаляет или добавляет активность или таймер относительно существующих открытых выполнений, движок реплея обнаруживает расхождение.

`WorkflowPanicPolicy` определяет поведение при обнаружении несоответствия:

```cpp
// По умолчанию — рекомендуется для продакшена:
opts.panic_policy = temporal::WorkflowPanicPolicy::BlockWorkflow;

// Терминальный вариант — используйте только когда застрявший воркфлоу хуже потерянного:
opts.panic_policy = temporal::WorkflowPanicPolicy::FailWorkflow;
```

- **`BlockWorkflow`** (по умолчанию): завершает *задачу* воркфлоу с ошибкой. Сервер повторит её, поэтому деплой исправленной сборки воркера может восстановить воркфлоу без потери данных. Это безопасный выбор, соответствующий поведению `BlockWorkflow` в Go/Java SDK.
- **`FailWorkflow`**: завершает *выполнение* воркфлоу с ошибкой окончательно. Терминально — используйте только когда застрявший воркфлоу хуже потерянного.

Полную модель обнаружения см. в [Обнаружение недетерминизма](/advanced#non-determinism-detection) в расширенном руководстве.

### Выявление проблем в CI с помощью реплей-тестирования

Самый надёжный способ предотвратить недетерминистичный деплой — воспроизвести реальные истории воркфлоу против изменённого кода до его выхода в продакшен. Класс `Worker` предоставляет `ReplayWorkflowHistory` именно для этой цели — он работает полностью офлайн, без необходимости в сервере:

```cpp
// 1. Экспортируйте представительную историю из продакшена или staging-запуска:
//    temporal workflow show -o json > fixtures/order-workflow.json
//    или:
std::string history_json = handle.FetchHistoryJson();  // из WorkflowHandle

// 2. В юнит-тесте воспроизведите её против текущего кода воркфлоу:
temporal::worker::Worker replayer(client, "replay-task-queue");
replayer.RegisterWorkflow("OrderWorkflow", OrderWorkflow);

// Бросает std::runtime_error, если воспроизведённые команды воркфлоу расходятся
// с записанной историей.
replayer.ReplayWorkflowHistory(history_json);
```

Добавьте несколько представительных историй как зафиксированные тестовые фикстуры и воспроизводите их в CI-пайплайне. Любое несовместимое изменение — переставленная активность, удалённый таймер, новая ветка, вставленная перед существующей — провалит тест вместо того, чтобы молча сломать выполняющийся воркфлоу в продакшене.

Полное описание см. в [Реплей-тестирование](/advanced#replay-testing) и [Тестирование](/testing).

## Плавное управление жизненным циклом воркера

`Worker` предоставляет три метода управления жизненным циклом:

```cpp
worker.Start();  // запустить потоки-поллеры — возвращает управление немедленно
// ...
worker.Stop();   // подать сигнал поллерам остановиться и дождаться завершения всех потоков
```

```cpp
worker.Run();    // Start() + блокировка до получения SIGINT/SIGTERM, затем Stop()
```

### Когда что использовать

**`Run()`** — правильная точка входа для выделенного процесса воркера. Он обрабатывает `SIGINT` и `SIGTERM` за вас и блокируется до получения запроса на завершение:

```cpp
int main() {
    auto client = temporal::client::Client::Connect(client_opts);

    temporal::worker::Worker worker(client, "orders", worker_opts);
    worker.RegisterWorkflow("OrderWorkflow", OrderWorkflow);
    worker.RegisterActivity("ChargeCard", ChargeCard);
    worker.RegisterActivity("SendEmail", SendEmail);

    worker.Run();   // блокируется; чистое завершение по SIGINT / SIGTERM
    return 0;
}
```

**`Start()` + `Stop()`** подходит для встраивания воркера в более крупный процесс, который самостоятельно управляет обработкой сигналов или последовательностью завершения:

```cpp
worker.Start();    // неблокирующий; поллеры запущены

// ... ваш сервер/основной цикл ...

// При завершении:
worker.Stop();     // сбросить активные задачи и дождаться завершения потоков-поллеров
```

:::note
`Stop()` подаёт поллерам сигнал прекратить принимать новые задачи и дожидается завершения потоков. Активности, уже взятые в работу, будут выполнены до конца (или до истечения `start_to_close_timeout`). Настраиваемого таймаута дрейна пока нет — `Stop()` блокируется до выхода всех поллеров.
:::

## Структурированное логирование

SDK никогда не пишет напрямую в `stdout`/`stderr`. Весь внутренний вывод проходит через `temporal::log::Logger`, который вы передаёте в `ClientOptions`. Один и тот же логгер используется воркером, движком реплея и слоем gRPC, так что вы получаете единый структурированный поток логов для всего SDK.

### Подключение своего логгера

```cpp
#include <temporal/log/logger.h>

class MyLogger : public temporal::log::Logger {
 public:
  void Log(temporal::log::Level level,
           std::string_view message,
           const std::vector<temporal::log::Field>& fields) override {
    // Передайте в slog, spdlog, absl::log или любой структурированный sink:
    auto* rec = my_log_library::NewRecord(to_severity(level), message);
    for (const auto& f : fields) {
      rec->AddField(f.key, f.value);
    }
    rec->Flush();
  }
};

temporal::ClientOptions opts;
opts.logger = std::make_shared<MyLogger>();

auto client = temporal::client::Client::Connect(opts);
// Воркер, созданный из этого клиента, наследует тот же логгер.
```

Если логгер не задан, используется `temporal::log::DefaultLogger()` — он пишет по одной записи на строку в `stderr`, что приемлемо для разработки.

### Уровни логирования и вспомогательные конструкторы

```cpp
namespace temporal::log {

enum class Level : std::uint8_t { Debug, Info, Warn, Error };

struct Field { std::string key; std::string value; };

// Вспомогательный конструктор:
Field F(std::string key, std::string value);

// Интерфейс логгера:
virtual void Log(Level level, std::string_view message,
                 const std::vector<Field>& fields) = 0;
}
```

SDK выдаёт сообщения уровня `Debug` для внутренних событий жизненного цикла (запуск поллера, диспетчеризация задач, попадания/промахи кэша), `Info` — для значимых изменений состояния (запуск/остановка воркера), `Warn`/`Error` — для неожиданных ситуаций. В продакшене подавляйте `Debug`, если не диагностируете конкретную проблему.

## Ещё не для продакшена {#not-yet-for-production}

:::warning
Следующие возможности **не реализованы**. Не полагайтесь на них в продакшене — они не деградируют корректно; их просто ещё нет.

| Функция | Статус | Влияние |
|---|---|---|
| TLS / mTLS | ❌ | Все соединения — незащищённый plaintext gRPC. Не выставляйте Temporal frontend в ненадёжную сеть без sidecar/proxy. |
| Аутентификация по API-ключу | ❌ | Воркеры и клиенты не могут аутентифицироваться в Temporal Cloud или защищённом self-hosted кластере. |
| Метрики (Prometheus / OTel) | ❌ | Нет метрик, эмитируемых SDK (задержка задач, загрузка слотов, schedule-to-start и т.д.). |
| Распределённая трассировка / OpenTelemetry | ❌ | Нет распространения трейсов через границы воркфлоу/активностей. |
| Версионирование воркеров / Build IDs | ❌ | Невозможно деплоить несколько версий воркеров в одну очередь задач с маршрутизацией. |
| Автомасштабирование поллеров / плавный дрейн | ❌ | Нет встроенного long-poll дрейна при `Stop()`; нет автомасштабирования. |
| Interceptors | ❌ | Нет middleware-хуков на клиенте или воркере. |
| Кодеки полезной нагрузки (шифрование/сжатие) | ❌ | Полезные нагрузки хранятся в истории Temporal как plain JSON. |

Полный список реализованного и нереализованного см. в [матрице совместимости](/parity).
:::

### Обходные пути для наиболее критичных пробелов

**Незащищённый транспорт**: запускайте воркеры `temporal-cpp-sdk` внутри сервисной сетки с mTLS-терминацией (Istio, Linkerd) или за Envoy sidecar, который завершает TLS перед передачей трафика на Temporal frontend.

**Нет метрик**: экспортируйте `cache_hits()` и `replays()` по периодическому таймеру в бэкенд метрик как минимальный показатель здоровья воркера до появления метрик на уровне SDK.

**Нет трассировки**: инструментируйте вход/выход активностей вручную с помощью предпочитаемой библиотеки трассировки; передавайте контекст трейса через memo воркфлоу или полезные нагрузки сигналов до реализации распространения заголовков/контекста.

## Собираем всё вместе

Минимальный готовый к продакшену процесс воркера, включающий все рекомендации этой страницы:

```cpp
#include <temporal/client/client.h>
#include <temporal/common/options.h>
#include <temporal/log/logger.h>
#include <temporal/worker/worker.h>

#include "my_logger.h"       // адаптер вашего структурированного логгера
#include "my_workflows.h"    // OrderWorkflow и др.
#include "my_activities.h"   // ChargeCard, SendEmail и др.

int main() {
    // --- Клиент ---
    temporal::ClientOptions client_opts;
    client_opts.target   = "temporal-frontend.internal:7233";  // только plaintext на данный момент
    client_opts.ns       = "production";
    client_opts.identity = "order-worker-v1";
    client_opts.logger   = std::make_shared<MyStructuredLogger>();

    auto client = temporal::client::Client::Connect(client_opts);

    // --- Воркер ---
    temporal::WorkerOptions worker_opts;
    worker_opts.max_concurrent_activities     = 100;
    worker_opts.max_concurrent_workflow_tasks = 20;
    worker_opts.activity_task_pollers         = 8;
    worker_opts.workflow_task_pollers         = 2;
    worker_opts.max_cached_workflows          = 500;
    // Оставляем BlockWorkflow (по умолчанию) — безопасно для rolling-деплоев.
    worker_opts.panic_policy = temporal::WorkflowPanicPolicy::BlockWorkflow;

    temporal::worker::Worker worker(client, "orders", worker_opts);
    worker.RegisterWorkflow("OrderWorkflow", OrderWorkflow);
    worker.RegisterActivity("ChargeCard",   ChargeCard);
    worker.RegisterActivity("SendEmail",    SendEmail);

    // Периодически отправляйте счётчики кэша в ваш бэкенд метрик.
    // (Встроенных метрик пока нет — см. «Ещё не для продакшена».)
    std::thread metrics_thread([&] {
        while (true) {
            std::this_thread::sleep_for(std::chrono::seconds(30));
            my_metrics::Gauge("temporal.cache_hits",  worker.cache_hits());
            my_metrics::Gauge("temporal.replays",     worker.replays());
        }
    });
    metrics_thread.detach();

    // Блокируется до SIGINT / SIGTERM, затем выполняет дрейн и завершается.
    worker.Run();
    return 0;
}
```
