# Тестирование

SDK поставляется с двумя тестовыми наборами: быстрые **модульные тесты** (без сервера) и
**интеграционные тесты**, выполняющие реальные воркфлоу против Temporal dev server.

## Модульные тесты

18 тестовых случаев на GoogleTest покрывают конвертер данных, примитив корутины (включая
раскрутку приостановленного стека при завершении) и шов среды исполнения воркфлоу (futures,
каналы сигналов, селекторы, отмена) в искусственном окружении. Сервер не требуется:

```bash
ctest --test-dir build -LE integration
```

## Интеграционные тесты

17 сквозных тестовых случаев выполняются против реального dev server и защищены флагом
`TEMPORAL_INTEGRATION=1`, так что запуск по умолчанию остаётся без сервера:

```bash
temporal server start-dev &
TEMPORAL_INTEGRATION=1 ctest --test-dir build -L integration
```

Они проверяют работу реального движка сквозным образом:

- таймер, одиночные и параллельные активности, распространение ошибок активности, быстрый отказ по `RetryPolicy`
- завершение (terminate), доставка сигналов и их упорядочивание, наблюдаемая отмена
- запрос (query) живого состояния, обновление (update) с накоплением состояния
- селектор (победа активности / победа по таймауту), дочерний воркфлоу
- продолжения sticky-кэша, heartbeat активности, цепочки continue-as-new
- офлайн-реплей истории (`ReplayWorkflowHistory`), повторное применение состояния обновления при реплее с нуля
- прерывание при дедлоке (перебравшая по времени задача проваливается и повторяется, пока воркер продолжает обслуживать другую работу)
- быстрая перемотка с пропуском времени (защищена `TEMPORAL_TEST_SERVER`; требует бинарник `temporal-test-server`)

## Тестовое окружение с пропуском времени

`temporal::testing::TestWorkflowEnvironment` подключается к **тестовому серверу Temporal с пропуском
времени** и быстро перематывает через ожидающие таймеры, поэтому логика с обилием таймеров
тестируется без ожиданий по настенным часам — воркфлоу, спящий «дни», завершается за миллисекунды,
а часы сервера прыгают вперёд на те же «дни».

:::warning
Для этого нужен **бинарник `temporal-test-server`** — отдельный процесс, *не* dev server, *не*
production-фронтенд и *не входящий в комплект* этого SDK. Запустите его сами заранее. RPC пропуска
времени (`GetCurrentTime` / `Sleep` / `LockTimeSkipping` / `UnlockTimeSkipping`) обслуживаются
только этим бинарником; их вызов против любого другого сервера выбрасывает `RpcError`
(`UNIMPLEMENTED`).
:::

```bash
# Запустите тестовый сервер с пропуском времени на выбранном вами порту:
temporal-test-server 7244
```

Тестовый сервер стартует с **заблокированным** пропуском времени (счётчик = 1). Конструирование
окружения подключается к нему и **разблокирует** пропуск времени (счётчик → 0), поэтому простаивающие
ожидающие таймеры пропускаются автоматически:

```cpp
#include <temporal/testing/test_workflow_environment.h>
#include <temporal/worker/worker.h>

// Подключается к уже запущенному тестовому серверу и разблокирует пропуск времени.
temporal::testing::TestWorkflowEnvironment env("localhost:7244");

// Постройте Worker (и запустите воркфлоу) на клиенте окружения:
temporal::worker::Worker worker(env.client(), "tq");
worker.RegisterWorkflow("Sleeper", Sleeper);   // тело вызывает ctx.Sleep(24h)
worker.Start();

const auto before = env.CurrentTime();
auto h = env.client().StartWorkflow(opts, "Sleeper");
h.Result<std::string>();                        // возвращает за мс — 24-часовой таймер пропущен
const auto skipped = env.CurrentTime() - before;  // ~24ч на (пропущенных) часах сервера
worker.Stop();
```

`env.client()` — это клиент, привязанный к тестовому серверу: запускайте на нём воркфлоу или
постройте `Worker` через `temporal::worker::Worker(env.client(), task_queue)`. Окружение также
предоставляет:

| Метод | Что он делает |
|---|---|
| `env.client()` | `client::Client`, привязанный к тестовому серверу. |
| `env.CurrentTime()` | Текущие (возможно, перемотанные вперёд) часы сервера как `system_clock::time_point`. |
| `env.Sleep(d)` | Блокировка до тех пор, пока часы сервера не продвинутся на `d` (перематывается, пока пропуск времени разблокирован). |
| `env.LockTimeSkipping()` / `env.UnlockTimeSkipping()` | Приостановить / возобновить автоматический пропуск. Блокировка **подсчитывается** — уравновешивайте каждую блокировку разблокировкой; время пропускается только когда счётчик равен `0`. |

Случай `TimeSkippingFastForwardsTimers` интеграционного набора проверяет это сквозным образом. Он
защищён `TEMPORAL_TEST_SERVER=host:port` и пропускается, когда эта переменная не задана, поэтому
обычный запуск против dev server не затрагивается:

```bash
temporal-test-server 7244 &
TEMPORAL_TEST_SERVER=localhost:7244 TEMPORAL_INTEGRATION=1 \
  ctest --test-dir build -L integration -R TimeSkipping
```

## Реплей записанных историй

`Worker::ReplayWorkflowHistory(history_json)` воспроизводит записанную историю воркфлоу против
зарегистрированного на воркере кода воркфлоу и обнаруживает недетерминизм — полностью **офлайн, без
контакта с сервером (без RPC)**. Он выбрасывает (`std::runtime_error`), если воспроизведённые
команды воркфлоу расходятся с записанной историей, что и есть способ поймать недетерминированное
изменение кода в модульном тесте до того, как оно сломает работающий воркфлоу в проде:

```cpp
// 1. Экспортируйте репрезентативную историю (зафиксированную как тестовая фикстура):
//    temporal workflow show -o json > fixtures/order-workflow.json
//    или из живого WorkflowHandle:
std::string history_json = handle.FetchHistoryJson();

// 2. Воспроизведите её против текущего кода воркфлоу:
temporal::worker::Worker replayer(client, "replay-task-queue");
replayer.RegisterWorkflow("OrderWorkflow", OrderWorkflow);
replayer.ReplayWorkflowHistory(history_json);  // выбрасывает при расхождении
```

См. [Запуск в проде](/production#catching-breakage-in-ci-with-replay-testing) об использовании
этого как CI-гейта.

### Обновления применяются повторно при реплее с нуля

Когда воркфлоу, принимавший **обновления**, воспроизводится с нуля — холодный старт, вытеснение из
sticky-кэша или `ReplayWorkflowHistory`, — принятые обработчики обновлений **выполняются заново в
своём записанном чередовании**. События, следующие за первым обновлением (разрешение future,
доставленный сигнал), откладываются и воспроизводятся в порядке истории с вставленными обратно на
исходные позиции обновлениями, так что любое состояние, изменённое обновлением (флаг, на котором тело
ждёт через `Await`, счётчик, который завершает более поздний сигнал), достоверно восстанавливается.
Маркеры / побочные эффекты *не* откладываются — они потребляются в порядке вызова. Интеграционный
случай `UpdateStateReappliedOnFullReplay` доказывает, что воркфлоу, чьи обновления в сумме дали `12`,
восстанавливает эти `12` после реплея на холодном воркере (было бы `0`, если бы обновления
отбрасывались). См. [Архитектуру](/architecture#replay-re-application-of-updates) о механизме.

## CI

`.github/workflows/ci.yml` запускает оба набора под macOS: устанавливает зависимости, собирает
проект, выполняет модульные тесты, затем поднимает dev server и прогоняет интеграционный набор.

## Замечание о ненадёжности тестов

Некоторые взаимодействия с Temporal являются *eventually consistent* (например, запрос (query),
отправленный сразу после сигнала). Тесты, читающие данные сразу после записи, опрашивают систему
до тех пор, пока ожидаемое состояние не станет видимым, а не делают одиночную проверку —
это правильный паттерн для eventually-consistent чтений.

## Что здесь отсутствует

**Окружение с пропуском времени** (выше) подключается к внешнему бинарнику `temporal-test-server`,
а не запускает мок-сервер в процессе — встроенного полностью in-memory тестового сервера нет.
Детерминированный **реплейер** (`Worker::ReplayWorkflowHistory`, выше) реализован и работает офлайн.
Смотрите [матрицу паритета](/parity) об оставшихся пробелах в поверхности тестирования.
