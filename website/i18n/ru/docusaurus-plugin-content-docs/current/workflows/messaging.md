# Сигналы, запросы и обновления

Три способа взаимодействовать с *работающим* воркфлоу — из клиента или из другого воркфлоу.

| Механизм      | Направление                           | Изменяет состояние? | Возвращает значение?   |
|---------------|---------------------------------------|---------------------|------------------------|
| **Сигнал**    | fire-and-forget в воркфлоу            | да                  | нет                    |
| **Запрос**    | запрос только для чтения              | нет                 | да (синхронно)         |
| **Обновление**| запрос + ответ                        | да                  | да (синхронно)         |

## Сигналы

Сигнал доставляет данные в работающий воркфлоу асинхронно. Воркфлоу считывает их из
**канала**:

```cpp
std::string GreetBySignal(temporal::workflow::Context& ctx) {
  auto channel = ctx.GetSignalChannel<std::string>("setName");
  std::string name = channel.Receive();   // блокируется (паркуется) до прихода сигнала
  return "Hello, " + name;
}
```

- `Receive()` блокируется до появления сигнала в буфере; `ReceiveAsync(out)` — неблокирующий вариант.
- Сигналы **буферизуются и упорядочиваются**: N-й вызов `Receive()` возвращает N-й сигнал с данным именем.

Отправка из клиента:

```cpp
auto dc = temporal::DataConverter::Default();
handle.Signal("setName", dc->ToPayloads(std::string("World")));
```

Распространённый паттерн — цикл, накапливающий сигналы:

```cpp
int Sum(temporal::workflow::Context& ctx) {
  auto channel = ctx.GetSignalChannel<int>("add");
  int total = 0;
  while (true) {
    int n = channel.Receive();
    if (n < 0) return total;   // сторожевое значение для завершения
    total += n;
  }
}
```

## Запросы (queries)

Запрос читает текущее состояние воркфлоу, не изменяя его. Зарегистрируйте обработчик в воркфлоу,
затем вызовите его из клиента:

```cpp
int Counter(temporal::workflow::Context& ctx) {
  int count = 0;
  ctx.SetQueryHandler("getCount", [&] { return count; });   // только чтение
  auto channel = ctx.GetSignalChannel<int>("add");
  while (true) count += channel.Receive();
}
```

```cpp
int n = handle.Query<int>("getCount");
```

- Обработчики запросов должны быть **только для чтения** (никаких активностей, таймеров или изменений состояния).
- Они выполняются на живом, приостановленном состоянии воркфлоу — именно поэтому данный SDK держит воркфлоу
  резидентным в [sticky-кэше](/architecture).
- Запросы обеспечивают согласованность по чтению после записи лишь *в конечном счёте* относительно только что отправленных сигналов; используйте опрос, если нужно наблюдать конкретную запись.

## Обновления (updates)

Обновление — это синхронный запрос/ответ, который **может изменять** состояние воркфлоу; по сути, это
запрос (query), результат которого также фиксируется в истории событий. Зарегистрируйте обработчик
и вызовите его из клиента:

```cpp
int Account(temporal::workflow::Context& ctx) {
  int balance = 0;
  ctx.SetUpdateHandler("deposit", [&](int amount) {
    balance += amount;
    return balance;          // возвращается вызывающей стороне
  });
  ctx.GetSignalChannel<std::string>("close").Receive();   // продолжаем работу
  return balance;
}
```

```cpp
int new_balance = handle.Update<int>("deposit", 100);   // блокируется до применения; возвращает 100
int again       = handle.Update<int>("deposit", 50);    // возвращает 150
```

Состояние накапливается между обновлениями, поскольку воркфлоу остаётся резидентным в sticky-кэше.

:::note Ограничения обновлений
**Валидаторы** обновлений (фаза предварительной валидации с возможностью отклонения) и **повторное
применение** принятых обновлений при реплее после вытеснения из кэша пока не реализованы. На практике
sticky-кэш удерживает работающий воркфлоу резидентным, поэтому обновления применяются по живому пути.
См. [матрицу паритета](/parity).
:::
