# Сигналы, запросы и обновления

Три способа взаимодействовать с *работающим* воркфлоу — из клиента или из другого воркфлоу.

| Механизм      | Направление                           | Изменяет состояние? | Возвращает значение?   |
|---------------|---------------------------------------|---------------------|------------------------|
| **Сигнал**    | fire-and-forget в воркфлоу            | да                  | нет                    |
| **Запрос**    | запрос только для чтения              | нет                 | да (синхронно)         |
| **Обновление**| запрос + ответ                        | да                  | да (синхронно)         |

## Сигналы

Сигнал доставляет данные в работающий воркфлоу асинхронно. Воркфлоу считывает их из
**канала**:

```cpp
std::string GreetBySignal(temporal::workflow::Context& ctx) {
  auto channel = ctx.GetSignalChannel<std::string>("setName");
  std::string name = channel.Receive();   // блокируется (паркуется) до прихода сигнала
  return "Hello, " + name;
}
```

- `Receive()` блокируется до появления сигнала в буфере; `ReceiveAsync()` возвращает
  `std::optional<T>` (либо используйте bool-форму `ReceiveAsync(out)`) как неблокирующий вариант.
- Сигналы **буферизуются и упорядочиваются**: N-й вызов `Receive()` возвращает N-й сигнал с данным именем.

Отправьте сигнал из клиента — вариативная перегрузка закодирует аргумент(ы) за вас:

```cpp
handle.Signal("setName", std::string("World"));   // кодирует через конвертер данных
```

Распространённый паттерн — цикл, накапливающий сигналы:

```cpp
int Sum(temporal::workflow::Context& ctx) {
  auto channel = ctx.GetSignalChannel<int>("add");
  int total = 0;
  while (true) {
    int n = channel.Receive();
    if (n < 0) return total;   // сторожевое значение для завершения
    total += n;
  }
}
```

## Запросы (queries)

Запрос читает текущее состояние воркфлоу, не изменяя его. Зарегистрируйте обработчик в воркфлоу,
затем вызовите его из клиента:

```cpp
int Counter(temporal::workflow::Context& ctx) {
  int count = 0;
  ctx.SetQueryHandler("getCount", [&] { return count; });   // только чтение
  auto channel = ctx.GetSignalChannel<int>("add");
  while (true) count += channel.Receive();
}
```

```cpp
int n = handle.Query<int>("getCount");
```

- Обработчики запросов должны быть **только для чтения** (никаких активностей, таймеров или изменений состояния).
- Они выполняются на живом, приостановленном состоянии воркфлоу — именно поэтому данный SDK держит воркфлоу
  резидентным в [sticky-кэше](/architecture).
- Запросы обеспечивают согласованность по чтению после записи лишь *в конечном счёте* относительно только что отправленных сигналов; используйте опрос, если нужно наблюдать конкретную запись.

## Обновления (updates)

Обновление — это синхронный запрос/ответ, который **может изменять** состояние воркфлоу; по сути, это
запрос (query), результат которого также фиксируется в истории событий. Зарегистрируйте обработчик
и вызовите его из клиента:

```cpp
int Account(temporal::workflow::Context& ctx) {
  int balance = 0;
  ctx.SetUpdateHandler("deposit", [&](int amount) {
    balance += amount;
    return balance;          // возвращается вызывающей стороне
  });
  ctx.GetSignalChannel<std::string>("close").Receive();   // продолжаем работу
  return balance;
}
```

```cpp
int new_balance = handle.Update<int>("deposit", 100);   // блокируется до применения; возвращает 100
int again       = handle.Update<int>("deposit", 50);    // возвращает 150
```

Состояние накапливается между обновлениями, поскольку воркфлоу остаётся резидентным в sticky-кэше,
а принятые обновления применяются повторно, если воркфлоу позднее воспроизводится из истории после
вытеснения из кэша.

### Валидаторы обновлений

Передайте третий аргумент в `SetUpdateHandler`, чтобы зарегистрировать **валидатор только для
чтения**. Он принимает те же аргументы, что и обработчик, и выполняется первым; если он выбрасывает
исключение, обновление отклоняется до принятия — ничего не записывается в историю, и обработчик не
выполняется:

```cpp
ctx.SetUpdateHandler(
    "deposit",
    [&](int amount) { balance += amount; return balance; },
    [](int amount) {
      if (amount <= 0) {
        throw temporal::ApplicationError("must be positive", "InvalidUpdate");
      }
    });
```

Валидаторы не должны изменять состояние или планировать активности/таймеры.

## Типизированные дескрипторы

Повторять имя на проводе в виде строки плюс явный шаблонный аргумент в каждой точке вызова —
чревато ошибками. Объявите типизированный дескриптор один раз на уровне неймспейса, и тип канала,
отправляемое значение и тип результата будут проверены и выведены автоматически:

```cpp
inline constexpr temporal::SignalRef<bool> kStop{"stop"};
inline constexpr temporal::QueryRef<int>   kSum{"sum"};
inline constexpr temporal::UpdateRef<int>  kBump{"bump"};
```

Внутри воркфлоу дескриптор заменяет имя (и `<T>` канала):

```cpp
ctx.SetQueryHandler(kSum, [&] { return sum; });
ctx.SetUpdateHandler(kBump, [&](int by) { sum += by; return sum; });
ctx.GetSignalChannel(kStop).Receive();   // ReceiveChannel<bool>, выведен
```

А из клиента типы значения и результата проверяются по ref:

```cpp
int after = h.Update(kBump, 5);   // аргумент + результат int выведены
int total = h.Query(kSum);        // результат int выведен
h.Signal(kStop, true);            // значение bool проверено по SignalRef<bool>
```

Типизированные дескрипторы сводятся к тем же строковым именам, поэтому взаимодействуют со строковым API выше.

## Сигнализация другому воркфлоу

Воркфлоу может отправить сигнал или отменить *другой* работающий воркфлоу по идентификатору
(fire-and-forget) — включая дочерний, который он запустил, передав идентификатор дочернего воркфлоу:

```cpp
ctx.SignalExternalWorkflow(target_id, "setName", std::string("World"));
ctx.CancelExternalWorkflow(target_id);
```

`SignalExternalWorkflow` кодирует свои аргументы через конвертер данных, как и вариативный `Signal`
клиента.

:::note Неизвестные идентификаторы
Вызовы со стороны клиента к идентификатору воркфлоу, который сервер не знает, выбрасывают
`temporal::NotFoundError` (подкласс `RpcError`, поэтому существующие места с `catch (RpcError&)`
по-прежнему его ловят).
:::
